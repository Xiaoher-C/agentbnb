  **CLI-01 — `init` command (line ~115):**
  After the existing local `bootstrapAgent(creditDb, owner, 100)` call (line ~164), add Registry grant logic:
  - Check if `existingConfig?.registry` (the registryUrl) is set
  - If yes, load identity auth via `loadIdentityAuth(owner)`, create a CreditLedger via `createLedger({ registryUrl: existingConfig.registry, ownerPublicKey: identityAuth.publicKey, privateKey: identityAuth.privateKey })`, then call `await ledger.grant(owner, 50)` (50cr per ADR — Registry grant is 50, local bootstrap stays 100 for backward compat)
  - Wrap in try/catch — if Registry is unreachable, warn but don't fail init
  - After grant, call `await ledger.getBalance(owner)` to get actual balance for display
  - Update console output: show "Registry balance: X credits" when registryUrl is set, keep "Credits: 100 (starter grant)" for local-only
  - Update JSON output similarly: add `registry_balance` field when applicable

  **CLI-02 — `status` command (line ~1053):**
  - Check `config.registry` — if set, create CreditLedger via `createLedger({ registryUrl: config.registry, ownerPublicKey, privateKey })` using `loadIdentityAuth(config.owner)`
  - Use `await ledger.getBalance(config.owner)` instead of `getBalance(creditDb, config.owner)`
  - Use `await ledger.getHistory(config.owner, 5)` instead of `getTransactions(creditDb, config.owner, 5)`
  - Keep the held escrows query as local-only (escrow table is still in local credit.db for local mode; for Registry mode, held escrows are tracked on Registry so this section can show "see Registry" or skip)
  - If no registryUrl, fall back to existing local SQLite logic (no change for local-only agents)

  **CLI-03 — `request` command (line ~730):**
  - For remote requests (isRemoteRequest=true) when config.registry is set:
    - Instead of calling `createSignedEscrowReceipt(creditDb, ...)` for local escrow, use CreditLedger: `const ledger = createLedger({ registryUrl: config.registry, ownerPublicKey, privateKey })` then `const { escrowId } = await ledger.hold(config.owner, amount, cardId)`
    - The settleEscrow helper should call `await ledger.settle(escrowId, targetOwner!)` instead of `settleRequesterEscrow(creditDb, escrowId)`
    - The releaseEscrow helper should call `await ledger.release(escrowId)` instead of `releaseRequesterEscrow(creditDb, escrowId)`
    - Keep existing local escrow path for non-Registry requests (isRemoteRequest=false or no config.registry)
  - Note: The relay already does credit hold/settle/release via Phase 28. For relay-routed requests, the CLI should NOT double-hold. Only hold locally when doing direct HTTP requests to providers who are NOT behind the relay. Check: if going through relay (tryViaRelay path), skip CLI-side escrow entirely — the relay handles it. If doing direct HTTP (requestCapability), use CreditLedger hold/settle/release.

  **CLI-04 — `publish` command (line ~349):**
  - After Zod validation succeeds (line ~394), check pricing:
    - For v2.0 cards: iterate `card.skills` and check each `skill.pricing.credits_per_call >= 1`
    - For v1.0 cards: check `card.pricing.credits_per_call >= 1`
    - If any skill has credits_per_call < 1, print error "Minimum price is 1 credit per call" and exit(1)
  - This applies to both local and remote publish

  <name>Task 2: Switch Hub /me and /me/transactions to CreditLedger</name>
  <files>src/registry/server.ts</files>
  <action>
  Import createLedger from '../credit/create-ledger.js' at the top.

  **HUB-01 — GET /me endpoint (line ~699):**
  - Currently: `const balance = opts.creditDb ? getBalance(opts.creditDb, ownerName) : 0;`
  - Change to: Create a CreditLedger from the creditDb using direct DB mode: `const ledger = createLedger({ db: opts.creditDb! })`. Then `const balance = await ledger.getBalance(ownerName)`.
  - This way the Registry server process uses direct DB mode (RegistryCreditLedger with mode: 'direct') — no HTTP round-trip to itself.
  - Keep the `opts.creditDb` guard — if no creditDb, balance stays 0.

  **HUB-02 — GET /me/transactions endpoint (line ~842):**
  - Currently: `const items = getTransactions(opts.creditDb, ownerName, limit);`
  - Change to: `const ledger = createLedger({ db: opts.creditDb! }); const items = await ledger.getHistory(ownerName, limit);`
  - Keep the guard: if no creditDb, return empty array.

  **HUB-03 — Zero frontend changes:**
  - The response shapes stay identical: `/me` returns `{ owner, balance }`, `/me/transactions` returns `{ items, limit }`.
  - CreditLedger.getHistory returns CreditTransaction[] which matches the existing getTransactions return type.
  - No changes to hub/ directory at all.

  **HUB-04 — OwnerDashboard displays real-time credit balance:**
  - This is satisfied by HUB-01 + HUB-03: the frontend fetches /me and gets the balance, which now comes from CreditLedger (Registry-backed on the server).

  **Performance note:** Creating a CreditLedger per request is cheap — it's just object construction with a DB reference, no connection pooling needed for direct mode. But if desired, create the ledger once at registration time (inside the `server.register(async (ownerRoutes) => { ... })` block) and reuse it across requests.
