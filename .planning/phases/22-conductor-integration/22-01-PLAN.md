Create `src/conductor/pipeline-orchestrator.ts` with:

```typescript
export interface OrchestrateOptions {
  subtasks: SubTask[];
  matches: Map<string, MatchResult>;
  gatewayToken: string;  // Token for auth with remote agents
  timeoutMs?: number;    // Per-task timeout, default 30000
}

export async function orchestrate(opts: OrchestrateOptions): Promise<OrchestrationResult>;
```

Implementation details:
1. Build a DAG from `subtasks[].depends_on` relationships.
2. Compute execution waves: tasks with no unresolved deps form a wave.
3. For each wave, execute all tasks in parallel via `Promise.allSettled()`.
4. For each task, call `requestCapability()` from `src/gateway/client.ts` with:
   - `gatewayUrl`: Derived from MatchResult.selected_agent (need to resolve agent -> URL, for now use params.gateway_url passed through or a resolver callback)
   - `cardId`: Use MatchResult's agent card
   - `params`: Merge subtask.params with outputs from completed dependency tasks
5. On failure, check `MatchResult.alternatives` and retry with the first alternative.
6. Track per-task spending and total credits.
7. Accumulate results in `Map<string, unknown>` keyed by subtask ID.
8. Return `OrchestrationResult` with aggregated results, total credits, timing, and any errors.

Important design decisions:
- Use a `resolveAgentUrl` callback parameter `(agentOwner: string) => { url: string; cardId: string }` to decouple from peer registry lookup. This allows E2E tests to inject mock URLs.
- Use `interpolate()` from `src/utils/interpolation.ts` for output piping: before executing step N, interpolate its params against a context object containing `{ steps: { [id]: result }, prev: lastResult }`.
- Do NOT import BudgetController here — budget checking is done by the caller (ConductorMode). PipelineOrchestrator is pure execution.

Tests: Use vi.mock to mock `requestCapability`. Create SubTask[] and MatchResult Map fixtures. Verify execution order, parallelism (check that Promise.allSettled is called with correct batch sizes), output piping, and retry behavior.

Create `src/conductor/conductor-mode.ts` implementing `ExecutorMode` interface from `src/skills/executor.ts`.

```typescript
export interface ConductorModeOptions {
  db: Database.Database;           // Registry DB for card search
  creditDb: Database.Database;     // Credit DB for budget checks
  conductorOwner: string;          // Self-exclusion for matching
  gatewayToken: string;            // Auth token for remote requests
  resolveAgentUrl: (owner: string) => { url: string; cardId: string };
  maxBudget?: number;              // Default 100
}

export class ConductorMode implements ExecutorMode {
  constructor(opts: ConductorModeOptions);
  async execute(config: SkillConfig, params: Record<string, unknown>): Promise<Omit<ExecutionResult, 'latency_ms'>>;
}
```

The execute() method:
1. Extract `task` string from `params.task`.
2. If skill is `orchestrate`:
   a. Call `decompose(task)` from task-decomposer.ts
   b. If empty result, return `{ success: false, error: "No template matches task" }`
   c. Call `matchSubTasks({ db, subtasks, conductorOwner })` from capability-matcher.ts
   d. Build MatchResult Map from array
   e. Create BudgetController with BudgetManager from creditDb
   f. Call `calculateBudget(matches)` then `canExecute(budget)`
   g. If budget not OK, return `{ success: false, error: "Budget exceeded: estimated X cr, max Y cr" }`
   h. Call `orchestrate({ subtasks, matches, gatewayToken })` from pipeline-orchestrator.ts
   i. Convert OrchestrationResult.results Map to plain object for JSON serialization
   j. Return `{ success: result.success, result: { plan, execution: resultObj, total_credits, latency_ms } }`
3. If skill is `plan`:
   a. Steps a-f same as orchestrate
   b. Return `{ success: true, result: { subtasks, matches, budget } }` without executing

Note: The SkillConfig param will have `type: 'conductor'`. This is a new executor type. It needs to be registered in the AgentRuntime modes map alongside api/pipeline/openclaw/command. However, registration in AgentRuntime is deferred to plan 22-02 (CLI wiring). For now, ConductorMode is a standalone module that implements ExecutorMode.
