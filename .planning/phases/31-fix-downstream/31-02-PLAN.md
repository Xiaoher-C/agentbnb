  <name>Task 2: Wire conduct with registryUrl and relay execution for remote agents</name>
  <files>src/cli/conduct.ts, src/conductor/pipeline-orchestrator.ts</files>
  <action>
    **Part A: conduct.ts — pass registryUrl to matchSubTasks**

    In `src/cli/conduct.ts`, find the `matchSubTasks()` call (around line 75). Add `registryUrl: config.registry` to the options:

    ```typescript
    matchResults = await matchSubTasks({
      db,
      subtasks,
      conductorOwner: config.owner,
      registryUrl: config.registry,  // NEW
    });
    ```

    Note: Plan 01 already made `matchSubTasks` async and updated this call to use `await`.

    **Part B: conduct.ts — update resolveAgentUrl to support relay for remote agents**

    The current `resolveAgentUrl` function (around line 115) only checks local peers and throws if unknown. For remote agents discovered via registry, they won't be in the local peers list. Update the function to:

    1. First try local peers (existing logic).
    2. If no local peer found AND `config.registry` is configured, return a sentinel URL indicating relay execution is needed. Use `relay://<owner>` as the URL convention.
    3. The MatchResult from remote search includes `selected_agent` (the remote agent's owner) — this is the relay target.

    **Part C: pipeline-orchestrator.ts — support relay execution**

    Currently `orchestrate()` calls `requestCapability()` for all tasks. For remote agents (where `resolveAgentUrl` returns a relay URL), it needs to use relay instead.

    Add optional `relayClient?: RelayClient` and `requesterOwner?: string` to `OrchestrateOptions`.

    In the execution loop where `requestCapability()` is called (around line 155), add a check:
    - If `primary.url.startsWith('relay://')` AND `opts.relayClient` is provided:
      - Extract target owner from URL: `primary.url.replace('relay://', '')`
      - Call `opts.relayClient.request({ targetOwner, cardId: primary.cardId, params: interpolatedParams, timeoutMs })` instead of `requestCapability()`
    - Same for alternative agent retry.

    **Part D: conduct.ts — pass relayClient when registry is configured**

    In conductAction, when `config.registry` is configured, create a temporary RelayClient for orchestration:

    ```typescript
    import { RelayClient } from '../relay/websocket-client.js';
    ```

    Before calling `orchestrate()`, check if config.registry exists. If so:
    1. Import identity auth: `const { publicKey, privateKey } = loadKeyPair()`
    2. Create a temporary RelayClient connected to the registry
    3. Pass it as `relayClient` to `orchestrate()`
    4. Disconnect the relay client after orchestration completes (in finally block)

    However, creating a full RelayClient with card registration just for conduct is heavyweight. Instead, use a simpler approach:

    **Simpler approach for relay execution in conduct.ts:**

    Instead of modifying pipeline-orchestrator.ts, override `resolveAgentUrl` in conduct.ts to handle relay execution inline. When the agent is remote (no local peer), use `requestViaRelay` from `src/gateway/client.ts` which already exists.

    Actually, looking at the architecture more carefully: `pipeline-orchestrator.ts` uses `requestCapability()` which is the HTTP gateway client. For relay execution, we need a different code path.

    **Final approach (minimal changes):**

    1. In `conduct.ts`, import `requestViaRelay` from `../gateway/client.js` (it's already exported).
    2. Modify `resolveAgentUrl` to return `{ url: 'relay://<owner>', cardId: '<card_id_from_match>' }` for remote agents.
    3. Add `relayClient` and `requesterOwner` to `OrchestrateOptions` in pipeline-orchestrator.ts.
    4. In orchestrate(), when URL starts with `relay://`, use relayClient.request() instead of requestCapability().
    5. In conduct.ts, when config.registry exists, lazily create a RelayClient, connect before orchestrate(), pass it, disconnect after.

    IMPORTANT: The RelayClient needs a card and onRequest handler to register. For the conduct use case, the conductor is a requester only (not a provider). Create it with a minimal card and no-op onRequest:

    ```typescript
    const relay = new RelayClient({
      registryUrl: config.registry,
      owner: config.owner,
      token: config.token ?? '',
      card: { id: config.owner, owner: config.owner, name: 'conductor' },
      onRequest: async () => ({ error: { code: -32601, message: 'Conductor does not accept requests' } }),
      silent: true,
    });
    await relay.connect();
    ```

    Pass `relay` to orchestrate as `relayClient`. In finally block: `relay.disconnect()`.
