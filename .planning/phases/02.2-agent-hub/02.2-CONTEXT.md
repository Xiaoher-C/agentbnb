<domain>
## Phase Boundary

Build a public, read-only capability browser page served at `/hub` on the existing registry port (7701). This is a cold-start accelerator for recruiting agent owners — anyone can browse what's available, creating "I want to put my agent up there too" motivation. Not a dashboard (private management UI).

</domain>

<decisions>
## Implementation Decisions

### Card visual design
- Visual hierarchy: Agent name (bold, largest) → Level badge → Category icon chips → Stats row (idle, success rate, credit)
- Identicon generated from agent_id using boring-avatars beam style — deterministic, zero-effort
- Keep card compact — no more than 5 lines of info
- Hover: subtle elevation (shadow lift) — signals clickability without visual noise
- Click: expand in-place to show full details (description, I/O schema, latency). No modal — user keeps spatial context of the grid
- Grid: CSS grid auto-fit with min-width ~320px — adapts naturally to screen width, no fixed column count

### Category mapping
- Category determined by Claude's discretion — likely infer from metadata.apis_used (maps known API names to categories) or metadata.tags. No schema change required.
- Cards with no category: show generic Puzzle icon + "custom" label
- Cards with multiple APIs: show all category chips (e.g. video pipeline shows Film + Volume2 + FileText)
- Max display 4 chips per card, overflow shows "+N more"
- Custom categories use Puzzle icon with custom category name as text
- Category filter dropdown: only show categories that currently have at least 1 card (dynamic, no empty filter results)

### Empty & loading states
- Loading: skeleton cards (gray rectangles pulsing)
- Empty (0 cards): friendly message "No agents online yet. Be the first — run `agentbnb init`" with copy-paste command block
- Error (registry unreachable): "Registry unreachable" with retry button
- Keep all states simple

### Header stats bar
- Always visible, even with zeros: "0 agents online | 0 capabilities | 0 exchanges"
- Shows data structure exists, honest about current state

### Hosting architecture (pre-decided)
- React SPA embedded in Fastify at `/hub` on port 7701
- Production: Fastify serves static build files
- Development: Vite proxy to Fastify API
- No separate server — reuses existing registry server

### Color palette
- Claude's discretion — should look "screenshot-worthy" as a recruiting tool

### Claude's Discretion
- Color palette choice (warm, cool, or dark — pick what looks best for a recruiting page)
- Category inference approach (apis_used vs tags vs hybrid)
- Typography scale and spacing
- Exact skeleton card design
- Loading/polling implementation details
- Card expand animation style

</decisions>

## Existing Code Insights

### Reusable Assets
- `createRegistryServer()` in `src/registry/server.ts` — standalone Fastify instance with CORS, GET /cards (paginated, filterable, sortable), GET /cards/:id
- `searchCards()` and `filterCards()` in `src/registry/matcher.ts` — FTS5 search + level/online filtering

### Established Patterns
- Fastify for all HTTP servers (gateway on 7700, registry on 7701)
- `agentbnb serve --registry-port 7701` starts registry alongside gateway
- ESM modules throughout, TypeScript strict mode
- Co-located tests as `*.test.ts`

### Integration Points
- Hub page will be served BY the registry server (same Fastify instance on 7701)
- Data source: existing GET /cards endpoint — no new backend routes needed
- `_internal` fields already stripped in /cards response
- `agentbnb serve` command in `src/cli/index.ts` will need to serve static hub files
