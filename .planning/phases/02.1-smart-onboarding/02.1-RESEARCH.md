# Phase 2.1: Smart Onboarding - Research

**Researched:** 2026-03-14
**Domain:** CLI onboarding UX, env var detection, process scanning, Capability Card draft generation
**Confidence:** HIGH

---

## Summary

Phase 2.1 adds sub-2-minute onboarding to `agentbnb init`. The core loop is: detect what APIs the
user already has configured (via env var presence only), optionally scan for running agent processes,
propose a draft Capability Card that maps detected APIs to known capability templates, and prompt the
user to confirm before publishing. Secondary goal is CLI ergonomics polish (better error messages,
colored output, progress indicators).

The entire implementation lives in Node.js standard APIs (`process.env`, `node:child_process`,
`node:os`, `node:readline`) plus the already-present Commander and Zod stack. No new runtime
dependencies are required. The security constraint is absolute: only `key in process.env` (boolean
existence) is ever evaluated — the value string is never read, logged, or written anywhere.

The draft Card generator is a pure mapping function: `detected_apis[] → CapabilityCard template`.
Templates are hard-coded in a lookup table (API name → level/inputs/outputs/pricing defaults). The
user reviews in an interactive confirm prompt before anything is written to disk. Non-interactive
environments (CI, `--yes` flag) skip the prompt and write directly.

**Primary recommendation:** Extend `agentbnb init` with a `--no-detect` escape hatch and an
interactive confirmation flow. Keep all new logic in `src/cli/onboarding.ts` (pure functions) so it
is independently testable without spawning the full CLI.

### Pattern 1: Env Var Existence Detection (Security-correct)
**What:** Check only the presence of known API key env vars, never their value.
**When to use:** Always — this is the only safe pattern given the security constraint.

```typescript
// Source: Node.js docs — process.env
// CORRECT: boolean existence check
const detected: string[] = [];
const KNOWN_API_KEYS = [
  'OPENAI_API_KEY',
  'ANTHROPIC_API_KEY',
  'ELEVENLABS_API_KEY',
  'KLING_API_KEY',
  'STABILITY_API_KEY',
  'REPLICATE_API_TOKEN',
  'GOOGLE_API_KEY',
  'AZURE_OPENAI_API_KEY',
  'COHERE_API_KEY',
  'MISTRAL_API_KEY',
] as const;

for (const key of KNOWN_API_KEYS) {
  if (key in process.env) {          // existence ONLY — value never read
    detected.push(key);
  }
}
```

**Never do:**
```typescript
// WRONG: reads the value
if (process.env['OPENAI_API_KEY']) { ... }
// WRONG: stores/logs the value
const val = process.env['OPENAI_API_KEY'];
```

### Pattern 2: Port Probe for Running Agents
**What:** Attempt a TCP connection to well-known agent ports; detect if anything is listening.
**When to use:** Scanning for running agent processes without needing root or ps parsing.

```typescript
// Source: Node.js net docs
import { createConnection } from 'node:net';

async function isPortOpen(port: number, host = '127.0.0.1', timeoutMs = 300): Promise<boolean> {
  return new Promise((resolve) => {
    const socket = createConnection({ port, host });
    const timer = setTimeout(() => { socket.destroy(); resolve(false); }, timeoutMs);
    socket.on('connect', () => { clearTimeout(timer); socket.destroy(); resolve(true); });
    socket.on('error', () => { clearTimeout(timer); resolve(false); });
  });
}

// Known agent/capability ports to probe:
const AGENT_PORTS = [7700, 7701, 8080, 3000, 8000, 11434 /* Ollama */];
```

**Confidence:** HIGH — Node.js net.createConnection is stable API.

### Pattern 4: Interactive Readline Confirm Prompt
**What:** Ask user to accept/edit/skip each draft card before publishing.
**When to use:** Interactive TTY sessions. Skip when `--yes` flag or `!process.stdout.isTTY`.

```typescript
// Source: Node.js readline docs
import { createInterface } from 'node:readline';

async function confirm(question: string): Promise<boolean> {
  const rl = createInterface({ input: process.stdin, output: process.stdout });
  return new Promise((resolve) => {
    rl.question(`${question} [y/N] `, (answer) => {
      rl.close();
      resolve(answer.toLowerCase().trim() === 'y');
    });
  });
}
```

### Pattern 5: Updated `init` Action Flow
```
agentbnb init [--owner NAME] [--port PORT] [--yes] [--no-detect]
  1. Detect env vars (unless --no-detect)
  2. Probe local ports (unless --no-detect)
  3. Summarize findings to stdout
  4. For each detected API:
     a. Render draft Capability Card to stdout
     b. If interactive && !--yes: confirm prompt
     c. If confirmed: publish to local registry
  5. Write config.json (existing behavior)
  6. Bootstrap credits (existing behavior)
  7. Print summary
```

### Anti-Patterns to Avoid
- **Reading env var values during detection:** `process.env['KEY']` in onboarding code — security violation
- **Blocking I/O for process scanning:** Use async port probes, not synchronous ps parsing
- **Mixing detection logic into `index.ts`:** Keep `onboarding.ts` pure so tests don't spawn subprocesses
- **Publishing without confirmation in interactive mode:** Always confirm when TTY, users need to review
- **Hard-coding card IDs:** Use `crypto.randomUUID()` for each draft card
- **Putting generated cards in config.json:** Cards go in registry SQLite only

## Common Pitfalls

### Pitfall 1: Reading Env Var Values
**What goes wrong:** Developer writes `if (process.env['OPENAI_API_KEY'])` — this reads the value
into the process heap, potentially visible in crash dumps or logs.
**Why it happens:** The idiomatic truthy check pattern (`if (process.env.FOO)`) is so common that
developers use it instinctively.
**How to avoid:** Enforce `key in process.env` exclusively. Add a comment at the top of
`onboarding.ts` explaining the constraint. Add a lint note or test that verifies the source doesn't
contain `process.env['OPENAI_API_KEY']` as a value read.
**Warning signs:** Any `process.env[key]` expression (index access returns the value string).

### Pitfall 2: Readline Not Closing on Early Exit
**What goes wrong:** If the user hits Ctrl+C during a confirm prompt, readline holds stdin open and
the process hangs.
**Why it happens:** `readline.createInterface` attaches to stdin; if not closed, the event loop stays
alive.
**How to avoid:** Always call `rl.close()` in both resolved and rejected paths. Alternatively wrap in
a `try/finally`.

### Pitfall 3: Port Probe Timing Out Too Slowly
**What goes wrong:** Probing 6 ports with a 5-second timeout each = 30 seconds before init completes.
**Why it happens:** Default TCP connect timeout on some systems is several seconds.
**How to avoid:** Use 300ms timeout per port, probe all ports in parallel via `Promise.all`.

```typescript
const results = await Promise.all(AGENT_PORTS.map(p => isPortOpen(p)));
```

### Pitfall 4: Draft Card Published Without Owner
**What goes wrong:** Generated card's `owner` field doesn't match the initialized agent's owner.
**Why it happens:** Template lookup table doesn't know the owner until init time.
**How to avoid:** Pass `owner` as a parameter into the card generator; never hardcode in templates.

### Pitfall 5: Non-TTY Environments Hanging on Readline
**What goes wrong:** In CI or piped contexts, `readline.question` never gets an answer because stdin
is not a terminal.
**Why it happens:** `readline.createInterface` reads from stdin — no human to type.
**How to avoid:** Check `process.stdout.isTTY` before entering interactive mode. If not TTY and no
`--yes` flag: skip draft card publishing (or default to skip with a notice).

### Pitfall 6: AGENTBNB_DIR Not Respected in Tests
**What goes wrong:** Tests that call detection functions write to real `~/.agentbnb/`.
**Why it happens:** `onboarding.ts` functions might call `getConfigDir()` — must pass through config
context, not call it directly.
**How to avoid:** `onboarding.ts` functions accept `configDir` as a parameter, never call
`getConfigDir()` internally. Tests pass a tmpDir.

## Code Examples

Verified patterns from Node.js built-ins (HIGH confidence):

### Existence-only env var detection
```typescript
// onboarding.ts
export function detectApiKeys(knownKeys: readonly string[]): string[] {
  return knownKeys.filter(key => key in process.env);
}
```

### Parallel port probing with timeout
```typescript
// onboarding.ts
export async function detectOpenPorts(ports: readonly number[]): Promise<number[]> {
  const results = await Promise.all(
    ports.map(async (port) => ({ port, open: await isPortOpen(port) }))
  );
  return results.filter(r => r.open).map(r => r.port);
}
```

### Draft card builder
```typescript
// onboarding.ts
import { randomUUID } from 'node:crypto';
import type { CapabilityCard } from '../types/index.js';

export function buildDraftCard(apiKey: string, owner: string): CapabilityCard | null {
  const template = API_TEMPLATES[apiKey];
  if (!template) return null;
  return {
    spec_version: '1.0',
    id: randomUUID(),
    owner,
    ...template,
    availability: { online: true },
    created_at: new Date().toISOString(),
    updated_at: new Date().toISOString(),
  };
}
```

### Interactive confirm with TTY guard
```typescript
// cli/index.ts (in init action)
const isInteractive = process.stdout.isTTY && !opts.yes;

for (const card of draftCards) {
  console.log(`\nDraft card: ${card.name}`);
  console.log(`  Level: L${card.level} | Credits: ${card.pricing.credits_per_call}/call`);
  console.log(`  APIs: ${card.metadata?.apis_used?.join(', ')}`);

  let publish = opts.yes;
  if (isInteractive) {
    publish = await confirm('Publish this card?');
  }

  if (publish) {
    const db = openDatabase(config.db_path);
    try { insertCard(db, card); } finally { db.close(); }
    console.log(`  Published: ${card.name} (${card.id.slice(0, 8)}...)`);
  } else {
    console.log('  Skipped.');
  }
}
```

## Open Questions

1. **Should multiple detected APIs produce a Pipeline (L2) card combining them?**
   - What we know: Templates currently generate one L1 card per API
   - What's unclear: Whether users prefer combined pipeline cards at init or individual atomics
   - Recommendation: Start with one L1 card per API — simpler to confirm, easier to edit later

2. **Should port probe results suggest `agentbnb connect` to the discovered agents?**
   - What we know: Port probing detects that something is listening, not that it's AgentBnB
   - What's unclear: How to distinguish an AgentBnB gateway from another HTTP server on port 8080
   - Recommendation: Send a GET /health request — AgentBnB gateway returns `{ status: 'ok' }`. If
     that matches, suggest `agentbnb connect`.

3. **AGENTBNB_DIR in test isolation for onboarding.ts**
   - What we know: `getConfigDir()` respects AGENTBNB_DIR env var
   - What's unclear: onboarding.ts needs `configDir` to write cards — should it call `getConfigDir()`
     or receive it as a parameter?
   - Recommendation: Receive as parameter. Tests inject a tmpDir. CLI passes `getConfigDir()`.
