PipelineOrchestrator does not currently call insertRequestLog — it returns results to its caller (ConductorMode). However, the team context (team_id + role per subtask) must flow through so callers can log it.

The correct approach: **do not add DB calls to pipeline-orchestrator.ts directly**. Instead, enrich the per-task result object returned from the inner map so that callers have team_id and role available for logging.

Specific changes:

1. The inner `async (taskId) =>` handler already returns `{ taskId, result, credits }`. Extend it to also return `team_id` and `role` when a team is present:

   After resolving `agentOwner` (line ~175):
   ```typescript
   // Capture team traceability context for the caller's log entry
   const teamMember = teamMemberMap.get(taskId);
   const teamId = opts.team?.team_id ?? null;
   const taskRole = teamMember?.role ?? null;
   ```

   Change the return on success from:
   ```typescript
   return { taskId, result: res, credits: m.credits };
   ```
   to:
   ```typescript
   return { taskId, result: res, credits: m.credits, team_id: teamId, role: taskRole };
   ```
   Apply the same change to the alternative agent success return:
   ```typescript
   return { taskId, result: altRes, credits: alt.credits, team_id: teamId, role: taskRole };
   ```

2. In the wave results collection loop (around line 241), update the destructuring:
   ```typescript
   const { taskId, result, credits, team_id, role } = settlement.value;
   ```
   Store team context alongside the result so the OrchestrationResult can carry it. Add `team_id` and `role` to what is stored in the `results` Map — but since the Map is `Map<string, unknown>`, the simplest approach is to store them in a parallel `traceContext` map:

   Inside the settlement collection:
