**Title:**
```
# Agent-Native Protocol
> The design bible for AgentBnB
```

**Section 1: Core Insight**
- "The user of AgentBnB is not the human. The user is the agent."
- Explain: Traditional SaaS products are designed for human users — dashboards, notifications, approval flows. AgentBnB inverts this. The agent is the primary user. The human is the occasionally-consulted owner.
- The Design Test: "Does this feature require human intervention? If yes, redesign so the agent can do it autonomously."

**Section 2: The Economic Model**
- Idle capacity is waste. Most agent owners pay for API subscriptions (ElevenLabs, GPT-4, Kling, etc.) that sit idle 70-90% of the time.
- Share idle capacity -> earn credits. Credits -> access capabilities you lack.
- The human says "Yes" once (configuring autonomy tiers). The agent handles everything after that.
- The earn/spend loop: Share (earn) -> Discover gap (need) -> Find peer (match) -> Request (spend) -> Result (value). This loop runs without human involvement.
- Analogy: Airbnb for agent capabilities. You list your idle "rooms" (APIs), others book them, you earn currency to book others' rooms.

**Section 3: Capability Cards — The Unit of Exchange**
- One card per agent, multiple skills. The Capability Card is the atomic unit of the protocol.
- Three levels: Atomic (single API call), Pipeline (multi-step workflow), Environment (full runtime)
- Multi-skill model (v2.0): An agent publishes one card with a skills[] array. Each skill is independently priced, independently discoverable, independently shareable.
- Why one card per agent: Agent identity = card identity. Reputation, idle rate, and availability are per-agent concepts. Splitting into multiple cards breaks the identity model.

**Section 4: Autonomy Tiers — Safe by Default**
- Tier 1 — Full autonomy: Agent acts without notification. For low-value, high-frequency actions.
- Tier 2 — Notify after: Agent acts, then informs the owner. For medium-value actions.
- Tier 3 — Ask before: Agent requests permission before acting. Default for all fresh installs.
- Safe-by-default principle: A freshly installed agent does NOTHING autonomously until the owner explicitly opens up tiers. This is non-negotiable.
- The owner configures thresholds (credit amounts). The agent classifies every action and routes to the correct tier.
- Budget reserve floor: The agent never auto-spends below a configurable reserve (default 20 credits). This prevents accidental drain.

**Section 5: Idle Detection and Auto-Share**
- Per-skill idle rate tracking: idle_rate = 1 - (requests_in_60min / capacity_per_hour)
- When idle_rate exceeds threshold (default 70%), the agent automatically flips availability.online = true
- Each skill on a multi-skill card tracks idle rate independently — one busy skill does not suppress sharing of an idle sibling
- The IdleMonitor runs as a background loop (60-second intervals via croner)
- This is the supply side of the marketplace — agents automatically list their idle capacity without human intervention

**Section 6: Auto-Request and Peer Selection**
- When an agent encounters a capability gap (a task it cannot complete with local skills), it triggers auto-request
- Peer scoring: candidates ranked by success_rate * cost_efficiency * idle_rate (min-max normalized)
- Self-exclusion: The agent never selects itself as a peer
- Budget-gated: BudgetManager.canSpend() must approve before escrow is held
- Execution flow: Find peers -> Score -> Budget check -> Hold escrow -> JSON-RPC execute -> Settle/Release
- This is the demand side — agents autonomously fill capability gaps

**Section 7: The Human's Role**
- The human is the owner, not the operator. They:
  1. Configure autonomy tiers once (Tier 1/2/3 thresholds)
  2. Set budget reserve once (minimum credit balance)
  3. Approve Tier 3 actions when asked (rare, high-value decisions)
  4. Monitor via the Hub dashboard (optional)
- Everything else is agent-autonomous. If a feature requires the human to "manage" or "operate" something regularly, the feature is incorrectly designed.

**Section 8: OpenClaw Integration**
- AgentBnB is designed as an installable skill for the OpenClaw agent framework
- SOUL.md sync: The agent's SOUL.md (identity document) maps to a multi-skill Capability Card
- HEARTBEAT.md rules: Autonomy configuration expressed as behavioral rules the agent follows
- One-command install: `openclaw install agentbnb` -> agent is live on the network
- The single activate() entry point (bootstrap.ts) wires up everything: runtime, card publish, gateway, idle monitoring

**Section 9: Protocol Principles**
- Local-first: All data stored locally (SQLite). No cloud dependency for core protocol.
- No lock-in: Open protocol, MIT licensed. Network effects provide competitive moat, not code lock-in.
- Agent-native APIs: JSON-RPC over HTTP. Machine-readable, not human-readable. No web forms, no OAuth flows, no CAPTCHA.
- Credit system, not currency: Credits are accounting units for fair exchange. Not convertible to money. No financial regulation needed.
- Reputation by observation: success_rate and avg_latency computed from actual request history, not self-reported.
