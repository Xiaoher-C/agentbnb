# Phase 2.25: Schema v1.1 Upgrade — Research

**Researched:** 2026-03-15
**Domain:** Zod schema extension, TypeScript type evolution, API field stripping, Hub React component updates
**Confidence:** HIGH

## Summary

Phase 2.25 is a targeted schema extension — three additive optional fields (`powered_by`, `_internal`, `free_tier`) on an existing Zod schema that already freezes at spec_version `'1.0'`. The schema extension is purely additive: no existing field changes, no migration needed for SQLite (cards stored as JSON blobs).

The critical implementation constraint is that `_internal` must be stripped from every API response path — the registry server's `GET /cards` and `GET /cards/:id` responses, and indirectly the Hub's useCards hook (which gets what the server sends). This stripping must happen server-side in `src/registry/server.ts`, not client-side.

The Hub already renders `powered_by` (the chain display was built in Phase 2.2). The only Hub changes needed are: adding `free_tier` to `HubCard` types and rendering a free-tier badge in the `CapabilityCard` component. No architectural changes are needed anywhere.

**Primary recommendation:** One plan is sufficient. All three field additions land in a single wave: schema + types + server stripping + Hub badge + tests. Estimated 8-12 files touched, all well-understood.

### Pattern 1: Additive Optional Zod Field
**What:** Add `.optional()` fields to existing `z.object()` without changing existing field constraints or spec_version.
**When to use:** Backward-compatible schema evolution where old cards remain valid.
**Example:**
```typescript
// Source: src/types/index.ts (current pattern to follow)
export const CapabilityCardSchema = z.object({
  // existing fields unchanged...
  powered_by: z.array(PoweredBySchema).optional(),  // already present
  // NEW fields to add:
  _internal: z.record(z.unknown()).optional(),
  // pricing sub-object change — inline:
  pricing: z.object({
    credits_per_call: z.number().nonnegative(),
    credits_per_minute: z.number().nonnegative().optional(),
    free_tier: z.number().nonnegative().optional(),  // ADD HERE
  }),
});
```

### Pattern 2: Server-Side Field Stripping
**What:** Delete a field from the response object before sending to caller.
**When to use:** Fields that are stored for internal use but must never leave the server boundary.
**Example:**
```typescript
// In src/registry/server.ts — apply before every reply.send(card) or reply.send(result)
function stripInternal(card: CapabilityCard): Omit<CapabilityCard, '_internal'> {
  const { _internal: _, ...public_card } = card;
  return public_card;
}

// Usage in GET /cards:
const items = cards.slice(offset, offset + limit).map(stripInternal);

// Usage in GET /cards/:id:
return reply.send(stripInternal(card));
```

### Pattern 3: Hub Free Tier Badge
**What:** Conditional badge rendering in `CapabilityCard.tsx` stats row.
**When to use:** Optional field — only render when `free_tier` is present and > 0.
**Example:**
```tsx
// In hub/src/components/CapabilityCard.tsx stats row (after credits_per_call display)
{card.pricing.free_tier !== undefined && card.pricing.free_tier > 0 && (
  <span className="text-emerald-400">{card.pricing.free_tier} free/mo</span>
)}
```

### Anti-Patterns to Avoid
- **Client-side _internal strip:** The Hub receives JSON from the server. Stripping in useCards or CapabilityCard.tsx is wrong — the server must never transmit it. Client-side stripping would be defense-in-depth at best, false security at worst.
- **Changing spec_version:** Do NOT change `spec_version` to `'1.1'`. The schema field is a `z.literal('1.0').default('1.0')` lock. Bumping it breaks all existing cards. These are additive optional fields within v1.0.
- **Breaking the pricing object shape:** `pricing` is used in `hub/src/lib/utils.ts` `formatCredits()` and in tests. Only add `free_tier` — do not restructure.
- **Stripping in the store layer:** `getCard()` and `searchCards()/filterCards()` in `store.ts` and `matcher.ts` should return raw cards. Stripping belongs at the API boundary in `server.ts`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Field stripping | Custom serializer / JSON replacer | Destructuring spread `{ _internal: _, ...rest }` | One line, type-safe, no extra deps |
| Schema extension | Schema migration / versioning system | `.optional()` on existing schema | Zod optional fields are backward-compat by definition |
| Hub type sync | Build-time codegen from src/types | Manual mirror in hub/src/types.ts | Hub is an independent Vite SPA — established pattern in this project |

**Key insight:** This phase is intentionally narrow. Every problem here has a 1-3 line solution using already-installed tools.

## Common Pitfalls

### Pitfall 1: _internal leaks through GET /cards/:id
**What goes wrong:** Developer strips `_internal` from the list endpoint (`GET /cards`) but forgets the single-card endpoint (`GET /cards/:id`), which calls `getCard()` directly and sends the raw result.
**Why it happens:** Two separate code paths in `server.ts` — one for list, one for single card (lines 97-169 vs lines 177-186).
**How to avoid:** Define `stripInternal()` as a shared helper at module scope, call it in both handlers. Test both endpoints explicitly.
**Warning signs:** Test for `_internal` absent from `GET /cards/:id` response — without this test, the bug survives.

### Pitfall 2: HubCard type out of sync with server response
**What goes wrong:** `free_tier` added to `CapabilityCardSchema` but not to `HubCard` interface in `hub/src/types.ts`. Hub TypeScript compiles but `card.pricing.free_tier` is silently `undefined` even when the server sends it.
**Why it happens:** Hub mirrors the schema manually (established decision: Hub is an independent SPA, no shared module). Two places must be updated in sync.
**How to avoid:** Update `HubCard.pricing` at the same time as the Zod schema. The type is small — easy to verify.

### Pitfall 3: free_tier renders as badge when value is 0
**What goes wrong:** `{card.pricing.free_tier && <Badge/>}` renders falsy when `free_tier = 0` (correctly, but unintentionally skips `free_tier = 0`). More importantly, `free_tier = 0` likely means "no free tier" — needs explicit `> 0` guard.
**Why it happens:** JavaScript falsy check on number conflates 0 with undefined.
**How to avoid:** Render badge only when `free_tier !== undefined && free_tier > 0`.

### Pitfall 4: powered_by tests already exist — don't duplicate
**What goes wrong:** Tests for `powered_by` were already written in `src/types/index.test.ts` (lines 131-190). Planner creates duplicate test suite.
**Why it happens:** powered_by was added in Phase 2.2 alongside the Hub work (see git commit `7946aab feat: add powered_by field to CapabilityCard schema`).
**How to avoid:** Research confirmed `powered_by` is already in the schema AND has full test coverage. Phase 2.25 needs only `_internal` and `free_tier` schema tests.

### Pitfall 5: spec_version must stay '1.0'
**What goes wrong:** Treating this as a "v1.1 upgrade" means bumping `spec_version` to `'1.1'`, which breaks `z.literal('1.0')` validation for all existing cards.
**Why it happens:** Phase name says "v1.1" but the implementation decision is additive-optional within the existing v1.0 schema contract.
**How to avoid:** Leave `spec_version: z.literal('1.0').default('1.0')` untouched. The fields are v1.1 in the changelog sense only — the wire format stays v1.0 compatible.

  it('accepts a card without free_tier (backward compat)', () => {

    expect(result.success).toBe(true);
  });

    const result = CapabilityCardSchema.safeParse(card);
    expect(result.success).toBe(true);
    if (result.success) expect(result.data.pricing.free_tier).toBe(100);

## Open Questions

1. **Should `_internal` be stripped from the CLI `discover` command output too?**
   - What we know: CLI `agentbnb discover` calls `searchCards()`/`filterCards()` on the local DB and formats output. This goes through a different path than the HTTP registry server.
   - What's unclear: If an agent publishes a card with `_internal` fields, does running `agentbnb discover` locally risk leaking those to stdout?
   - Recommendation: Strip `_internal` at the CLI output layer too, in `src/cli/index.ts` discover handler. Low effort, consistent behavior.

2. **Does `_internal` need to be documented as a stripping contract?**
   - What we know: The schema field is named with `_` prefix to signal private. Stripping is implemented in server.ts.
   - What's unclear: No JSDoc currently explains the strip contract.
   - Recommendation: Add a JSDoc comment on the field in the Zod schema explaining that it is stripped from all API and CLI responses.
