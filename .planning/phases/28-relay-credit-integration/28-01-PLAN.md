<objective>
Wire credit hold/settle/release into the WebSocket relay request flow so that every relayed capability request has credits escrowed before the provider receives it.

Purpose: Without credit integration, agents can use capabilities without paying. This plan makes the relay enforce payment at the protocol level — hold before forwarding, settle on success, release on failure/timeout/disconnect.

Output: Modified relay with credit operations, a relay-credit helper module, and tests proving the hold/settle/release flow.
</objective>

    2. **Modify `handleRelayRequest`** — after the rate limit check and before forwarding to the target agent, insert credit hold logic:
       - If `creditDb` is undefined, skip credit operations (backward compat for tests without credits).
       - Call `lookupCardPrice(db, msg.card_id, msg.skill_id)` to get the price.
       - If price is null or 0, proceed without escrow (free skill).
       - If price > 0, call `holdForRelay(creditDb, fromOwner, price, msg.card_id)`.
       - If hold throws `INSUFFICIENT_CREDITS`, send an error response to the requester with `{ type: 'response', id: msg.id, error: { code: -32603, message: 'Insufficient credits' } }` and return early (do NOT forward to provider).
       - On successful hold, store the escrowId in the PendingRelayRequest: `pendingRequests.set(msg.id, { originOwner: fromOwner, timeout, escrowId, targetOwner: msg.target_owner })`.

    3. **Modify `handleRelayResponse`** — after clearing the timeout and before forwarding the response:
       - If `pending.escrowId` exists and `creditDb` is defined:
         - If `msg.error` is undefined (success): call `settleForRelay(creditDb, pending.escrowId, pending.targetOwner!)` — the targetOwner is the provider who should receive credits.
         - If `msg.error` is defined (provider returned an error): call `releaseForRelay(creditDb, pending.escrowId)` — refund the requester.

    4. **Modify timeout handler** in `handleRelayRequest` — when the timeout fires:
       - If `pending.escrowId` exists and `creditDb` is defined: call `releaseForRelay(creditDb, pending.escrowId)`.
       - Same for the timeout handler in `handleRelayProgress`.

    5. **Modify `handleDisconnect`** — when a provider disconnects, iterate pendingRequests and release escrow for any pending request where `pending.targetOwner === owner`:
       - For each such pending request: if `pending.escrowId` and `creditDb`: call `releaseForRelay(creditDb, pending.escrowId)`. Also clear the timeout and send an error response to the originOwner: `{ type: 'response', id: reqId, error: { code: -32603, message: 'Provider disconnected' } }`.
       - Delete from pendingRequests.

    6. **Modify `registerWebSocketRelay` in server.ts** — pass `opts.creditDb` as the third argument:


    7. **Wrap credit operations in try/catch** within the relay — credit failures (DB errors) should not crash the relay. Log errors but still forward/respond. The only hard rejection is INSUFFICIENT_CREDITS on hold.
