# Phase 2.3: Remote Registry Discovery - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add `--registry <url>` support to `agentbnb discover` so the CLI can query remote registry servers via HTTP, completing the cross-machine discovery loop that Phase 2's registry API created. Also add a `config` subcommand for persistent settings and a `--tag` filter flag.

</domain>

<decisions>
## Implementation Decisions

### Registry URL handling
- `--registry <url>` flag is optional on the `discover` command
- If not provided, check `~/.agentbnb/config.json` for a saved `registry` default
- If neither exists, use local DB only (current behavior)
- New `agentbnb config set <key> <value>` / `agentbnb config get <key>` subcommand for persistent settings
- `agentbnb config set registry https://registry.agentbnb.dev` saves default registry URL
- When both local DB and remote registry are available, merge results — deduplicated by card ID (local version wins)

### Filter forwarding
- Forward existing CLI filters to remote API: q, level, online, tag
- Add new `--tag <tag>` flag to the `discover` command (registry API already supports it)
- Do NOT add advanced filters (min_success_rate, max_latency_ms, sort, limit, offset) yet — keep minimal for v1
- If remote API doesn't support a filter, apply it client-side after fetching

### Error UX
- 5-second timeout for remote registry HTTP requests
- Fail fast, no retry — users retry manually
- Error messages per type:
  - Timeout: "Registry at {url} did not respond within 5s. Showing local results only."
  - Connection refused: "Cannot reach {url}. Is the registry running? Showing local results only."
  - Invalid URL: "Invalid registry URL: {url}"
  - Auth error: "Authentication failed for {url}. Run `agentbnb config set token <your-token>`."
- When `--registry` flag explicitly provided and fails: fail with error
- When using saved config default and fails: show local results + warning

### Output merging
- Merged results displayed as one unified list — no visual separation between local/remote sections
- Small `[local]` or `[remote]` tag at end of each table row for source transparency
- `--json` output adds `"source": "local"` or `"source": "remote"` field to each card object
- When searching with a query: interleave results by relevance regardless of source
- When browsing without query: local first, then remote (natural for agent owner perspective)
- Deduplication by card UUID — local version wins when same card exists in both

### Claude's Discretion
- HTTP client library choice (native fetch, undici, or node-fetch)
- Relevance interleaving algorithm when merging search results
- Config file format details (extend existing config.json or separate file)
- `config` subcommand help text and validation

</decisions>

<specifics>
## Specific Ideas

- Error messages should suggest actionable next steps (e.g., "Is the registry running?", "Run `agentbnb config set token`")
- The `config` subcommand is generic and extensible — not registry-specific. Future settings (token, default port, etc.) use the same mechanism
- Client-side filter fallback pattern ensures CLI upgrades don't depend on API upgrades

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `discover` command in `src/cli/index.ts:258-351` — already has --level, --online, --local, --json flags and table/JSON output formatting
- `createRegistryServer()` in `src/registry/server.ts` — GET /cards endpoint with q, level, online, tag, min_success_rate, max_latency_ms, sort, limit, offset params
- `loadConfig()` in CLI — reads ~/.agentbnb/config.json, returns config object with db_path, owner, host, port
- `searchCards()` / `filterCards()` in `src/registry/matcher.ts` — local query functions

### Established Patterns
- `AGENTBNB_DIR` env var for config dir override (test isolation)
- Commander-based CLI with subcommands, --help, --json output
- Fastify for HTTP servers, co-located tests as *.test.ts
- `col()` helper for table column formatting in discover output

### Integration Points
- `discover` command action handler needs a new branch for remote fetch (alongside existing local DB and --local mDNS branches)
- Config file at `getConfigDir()/config.json` — needs `registry` field added
- New `config` command registers on the Commander `program` instance in `src/cli/index.ts`
- GET /cards API at `<registry-url>/cards` is the remote data source

</code_context>

<deferred>
## Deferred Ideas

- Advanced remote filters (min_success_rate, max_latency_ms, sort, pagination) — add when users need marketplace-level browsing from CLI
- Registry authentication (token-based) — add when private registries exist
- Multiple registry sources (query several registries in parallel) — future scalability concern

</deferred>
