## User Constraints (from CONTEXT.md)

### Locked Decisions

**Registry URL handling**
- `--registry <url>` flag is optional on the `discover` command
- If not provided, check `~/.agentbnb/config.json` for a saved `registry` default
- If neither exists, use local DB only (current behavior)
- New `agentbnb config set <key> <value>` / `agentbnb config get <key>` subcommand for persistent settings
- `agentbnb config set registry https://registry.agentbnb.dev` saves default registry URL
- When both local DB and remote registry are available, merge results — deduplicated by card ID (local version wins)

**Filter forwarding**
- Forward existing CLI filters to remote API: q, level, online, tag
- Add new `--tag <tag>` flag to the `discover` command (registry API already supports it)
- Do NOT add advanced filters (min_success_rate, max_latency_ms, sort, limit, offset) yet — keep minimal for v1
- If remote API doesn't support a filter, apply it client-side after fetching

**Error UX**
- 5-second timeout for remote registry HTTP requests
- Fail fast, no retry — users retry manually
- Error messages per type:
  - Timeout: "Registry at {url} did not respond within 5s. Showing local results only."
  - Connection refused: "Cannot reach {url}. Is the registry running? Showing local results only."
  - Invalid URL: "Invalid registry URL: {url}"
  - Auth error: "Authentication failed for {url}. Run `agentbnb config set token <your-token>`."
- When `--registry` flag explicitly provided and fails: fail with error
- When using saved config default and fails: show local results + warning

**Output merging**
- Merged results displayed as one unified list — no visual separation between local/remote sections
- Small `[local]` or `[remote]` tag at end of each table row for source transparency
- `--json` output adds `"source": "local"` or `"source": "remote"` field to each card object
- When searching with a query: interleave results by relevance regardless of source
- When browsing without query: local first, then remote (natural for agent owner perspective)
- Deduplication by card UUID — local version wins when same card exists in both

### Claude's Discretion
- HTTP client library choice (native fetch, undici, or node-fetch)
- Relevance interleaving algorithm when merging search results
- Config file format details (extend existing config.json or separate file)
- `config` subcommand help text and validation

### Deferred Ideas (OUT OF SCOPE)
- Advanced remote filters (min_success_rate, max_latency_ms, sort, pagination)
- Registry authentication (token-based)
- Multiple registry sources (query several registries in parallel)

## Summary

Phase 2.3 is a focused CLI extension. The codebase already has every infrastructure piece: native `fetch` with `AbortController` is established in `src/gateway/client.ts`, the registry HTTP server exists (`GET /cards` with full filter support), the discover command has table/JSON output, and the CLI test harness (`runCli()` + `AGENTBNB_DIR`) is proven across 200 tests.

The implementation is three discrete additions: (1) a `fetchRemoteCards()` function mirroring the gateway client pattern, (2) a `config` subcommand wired to `loadConfig()`/`saveConfig()` with a `registry` field added to `AgentBnBConfig`, and (3) updated `discover` command action that merges local + remote results with dedup and `[local]`/`[remote]` source tags.

Testing follows the server.test.ts in-process injection pattern for unit tests, and a new integration test in `index.test.ts` that starts a real Fastify server on port 0 (OS-assigned ephemeral port) before running the CLI subprocess — the established pattern for cross-process integration testing in this codebase.

**Primary recommendation:** Use native Node.js `fetch` + `AbortController` (already proven in `gateway/client.ts`). Extend `AgentBnBConfig` with an optional `registry?: string` field. Write `fetchRemoteCards()` in a new `src/cli/remote-registry.ts` file following the client.ts style. The `config` subcommand operates on the same `config.json` via `loadConfig()`/`saveConfig()`.

### Pattern 1: Remote Fetch with AbortController Timeout

**What:** Fetch remote `/cards` with forwarded query params, classify errors by type.
**When to use:** Whenever `--registry <url>` is provided or `config.registry` is set.
**Example:**

```typescript
// Source: gateway/client.ts (established pattern, adapted for GET /cards)
export async function fetchRemoteCards(
  registryUrl: string,
  params: { q?: string; level?: number; online?: boolean; tag?: string },
  timeoutMs = 5_000
): Promise<CapabilityCard[]> {
  const url = new URL('/cards', registryUrl);
  if (params.q) url.searchParams.set('q', params.q);
  if (params.level !== undefined) url.searchParams.set('level', String(params.level));
  if (params.online !== undefined) url.searchParams.set('online', String(params.online));
  if (params.tag) url.searchParams.set('tag', params.tag);

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeoutMs);

  let response: Response;
  try {
    response = await fetch(url.toString(), { signal: controller.signal });
  } catch (err) {
    clearTimeout(timer);
    const isTimeout = err instanceof Error && err.name === 'AbortError';
    if (isTimeout) throw new RegistryTimeoutError(registryUrl);
    throw new RegistryConnectionError(registryUrl);
  } finally {
    clearTimeout(timer);
  }

  if (response.status === 401 || response.status === 403) {
    throw new RegistryAuthError(registryUrl);
  }

  const body = await response.json() as { items: CapabilityCard[] };
  return body.items;
}
```

### Pattern 3: Result Merging with Dedup and Source Tagging

**What:** Merge local and remote card arrays, deduplicate by `id` (local wins), tag each with source.
**When to use:** After fetching both local and remote results.
**Example:**

```typescript
type TaggedCard = CapabilityCard & { source: 'local' | 'remote' };

function mergeResults(
  localCards: CapabilityCard[],
  remoteCards: CapabilityCard[],
  hasQuery: boolean
): TaggedCard[] {
  const tagged: TaggedCard[] = [
    ...localCards.map(c => ({ ...c, source: 'local' as const })),
    ...remoteCards.map(c => ({ ...c, source: 'remote' as const })),
  ];

  // Dedup: local wins — keep first occurrence (local is prepended)
  const seen = new Set<string>();
  const deduped = tagged.filter(c => {
    if (seen.has(c.id)) return false;
    seen.add(c.id);
    return true;
  });

  if (hasQuery) {
    // Interleave by relevance: alternate local/remote from deduplicated lists
    // Simple approach: stable interleave (alternating zip) preserves per-source ordering
    const locals = deduped.filter(c => c.source === 'local');
    const remotes = deduped.filter(c => c.source === 'remote');
    const interleaved: TaggedCard[] = [];
    const maxLen = Math.max(locals.length, remotes.length);
    for (let i = 0; i < maxLen; i++) {
      if (i < locals.length) interleaved.push(locals[i]);
      if (i < remotes.length) interleaved.push(remotes[i]);
    }
    return interleaved;
  }

  // Browsing: local first (already the order after dedup since local is prepended)
  return deduped;
}
```

### Pattern 5: Integration Test with Real Server

**What:** Start a real Fastify registry server, get ephemeral port, run CLI subprocess pointing at it.
**When to use:** RRD-02 integration test.
**Example:**

```typescript
// src/cli/index.test.ts — new integration describe block
import { createRegistryServer } from '../registry/server.js';
import { openDatabase, insertCard } from '../registry/store.js';

describe('CLI: discover --registry (integration)', () => {
  let tmpDir: string;
  let registryServer: ReturnType<typeof createRegistryServer>;
  let registryPort: number;
  let db: Database.Database;

  beforeEach(async () => {
    tmpDir = mkdtempSync(join(tmpdir(), 'agentbnb-test-'));
    runCli('init --owner remote-test-agent', tmpDir);

    db = openDatabase(':memory:');
    insertCard(db, makeCard('remote-card-uuid', 'Remote Capability'));

    registryServer = createRegistryServer({ registryDb: db, silent: true });
    const address = await registryServer.listen({ port: 0, host: '127.0.0.1' });
    registryPort = new URL(address).port ? parseInt(new URL(address).port) : 0;
    // Alternative: registryServer.server.address() as AddressInfo
  });

  afterEach(async () => {
    await registryServer.close();
    db.close();
    rmSync(tmpDir, { recursive: true, force: true });
  });

  it('discovers cards from remote registry via --registry flag', () => {
    const { status, stdout } = runCli(
      `discover --registry http://127.0.0.1:${registryPort}`,
      tmpDir
    );
    expect(status).toBe(0);
    expect(stdout).toContain('Remote Capability');
    expect(stdout).toContain('[remote]');
  });

  it('shows error and local results when registry unreachable', () => {
    const { status, stdout, stderr } = runCli(
      'discover --registry http://127.0.0.1:19999',
      tmpDir
    );
    // When using config default, should degrade gracefully
    expect(stdout).toContain('Cannot reach');
  });
});
```

**Note on Fastify listen return value:** `server.listen()` resolves with the address string (e.g., `http://127.0.0.1:PORT`) in Fastify v5. The ephemeral port can be extracted via `new URL(address).port` or `(server.server.address() as import('net').AddressInfo).port`.

### Anti-Patterns to Avoid

- **Modifying table column widths for non-remote runs:** Only add the Source column when remote sources are actually involved. Pure local `discover` should preserve the existing layout (avoid regressions in existing tests).
- **Throwing on config default failures:** When a registry URL comes from saved config (not explicit `--registry` flag), timeout/connection errors must degrade gracefully — show local results + warning, do not exit(1).
- **Re-validating remote cards through Zod:** Remote cards are already validated by the server. Parsing them through CapabilityCardSchema client-side will fail if schema versions diverge. Accept them as `unknown` and cast, or use `.partial()`.
- **Hard-coding `limit` in remote fetch:** The GET /cards endpoint defaults to 20 items. For CLI use, pass `limit=100` (the server-side cap) to avoid pagination truncation without implementing full pagination.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP timeout | Custom polling/timer loop | `AbortController` + `setTimeout` | Already proven in gateway/client.ts; AbortController is the standard Web API approach |
| Query string building | Manual string concat | `URLSearchParams` (built-in) | Handles encoding, undefined values, duplicate keys correctly |
| URL validation | Regex or manual parse | `new URL(url)` in try/catch | Throws on invalid URLs; catches malformed input before the fetch attempt |
| Ephemeral port in tests | Fixed port (race conditions) | `listen({ port: 0 })` | OS assigns free port; eliminates EADDRINUSE flakiness in CI |

**Key insight:** The gateway client is the canonical reference implementation for all HTTP patterns in this codebase. `fetchRemoteCards()` should be a structural copy of `requestCapability()` adapted for GET.

## Common Pitfalls

### Pitfall 1: Invalid URL crashes instead of user-friendly error

**What goes wrong:** `new URL(url)` throws a `TypeError` synchronously if the URL is malformed. If not caught, the CLI crashes with an unformatted error instead of the specified message "Invalid registry URL: {url}".
**Why it happens:** Developers validate URL format inside the `fetch()` try/catch, but `new URL()` throws before `fetch()` is called.
**How to avoid:** Validate the URL first in a dedicated `try { new URL(url) } catch { throw new InvalidRegistryUrlError(url) }` block before constructing the request.
**Warning signs:** Tests for invalid URLs exit with status 1 but show a Node.js TypeError stack trace rather than the expected message.

### Pitfall 2: Existing discover tests break because table width changed

**What goes wrong:** Adding a Source column increases line width. Existing tests that check for exact output strings (e.g., `expect(stdout).toContain('Alpha TTS')`) may still pass, but tests checking column alignment or line length fail.
**Why it happens:** The table header/separator line width is hardcoded (`'-'.repeat(72)`).
**How to avoid:** Only emit the Source column when remote sources are included. Local-only `discover` (no `--registry` and no `config.registry`) renders the existing table unchanged.
**Warning signs:** The 5 existing discover tests in `index.test.ts` fail after the change.

### Pitfall 3: Remote cards missing `_internal` field cause TypeScript narrowing issues

**What goes wrong:** Remote cards arrive from the API without `_internal` (server strips it). The local `stripInternal` destructure `({ _internal: _, ...rest })` will silently no-op on remote cards, but strict TypeScript may warn about destructuring a field that may not exist.
**Why it happens:** `AgentBnBConfig` has `_internal` as optional; the destructure pattern is safe at runtime but may produce TS lint warnings.
**How to avoid:** Apply `stripInternal` consistently to all cards (local and remote) before output. This is already how local cards are handled.

### Pitfall 4: Fastify listen() return value differs between v4 and v5

**What goes wrong:** In Fastify v4, `listen()` returns a string address. In Fastify v5 (used here), it still returns a string, but the format may differ. Extracting the port via regex is brittle.
**Why it happens:** Integration tests need the actual assigned port after `listen({ port: 0 })`.
**How to avoid:** Use `(server.server.address() as import('net').AddressInfo).port` — the underlying `net.Server` always provides this after listen.
**Warning signs:** Port extracted from listen() returns `0` or `NaN`.

### Pitfall 5: Config `set` with unknown keys silently saves garbage

**What goes wrong:** `agentbnb config set typo_key value` saves a key that nothing reads, giving users false confidence.
**Why it happens:** The generic key-value interface accepts any string.
**How to avoid:** Validate against a known-keys allowlist: `['registry', 'token']`. Reject unknown keys with an error. This is Claude's discretion — implement validation.
