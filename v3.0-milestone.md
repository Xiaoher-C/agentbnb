# AgentBnB v3.0 Milestone — Production-Ready Launch

> Claude Code: read AGENT-NATIVE-PROTOCOL.md and docs/brain/00-MOC.md first, then this file.
> This milestone addresses the CRITICAL architecture gaps that prevent AgentBnB from functioning as a real P2P capability exchange network.

---

## Context

After 62+ plans across 5 milestones (v1.1 → v2.3), AgentBnB has:
- ✅ Communication layer (Gateway, auth, JSON-RPC)
- ✅ Detection layer (IdleMonitor, per-skill idle rate)
- ✅ Decision layer (AutoRequestor, peer scoring, autonomy tiers)
- ✅ Budgeting layer (BudgetManager, escrow, reserve floor)
- ✅ UI layer (Hub with 5 pages, Magic UI components, dark theme)
- ✅ Distribution layer (Claude Code plugin, OpenClaw skill, SKILL.md)

But it CANNOT actually do anything. Two critical gaps:
1. **No execution engine** — Gateway dispatches to empty localhost:8080
2. **No cross-machine credits** — P2P credit verification fails across separate SQLite DBs

This milestone fixes both gaps, adds the Conductor orchestrator agent, and deploys to production.

## Phase 19: SkillExecutor — Agent Self-Execution Engine (Track A)

### Why This Is Priority #0

The current architecture has every layer EXCEPT execution. When Gateway receives a request, it dispatches to `--handler-url` (default localhost:8080) — but nothing listens there. Without SkillExecutor, agents are communication pipes with nothing flowing through them.

### Architecture Addition

```
AgentRuntime (existing)
├── Gateway          (communication — exists)
├── IdleMonitor      (detection — exists)
├── AutoRequestor    (requesting — exists)
├── BudgetManager    (budgeting — exists)
└── SkillExecutor    (execution — NEW)
    ├── APIExecutor        (call external APIs via config)
    ├── PipelineExecutor   (chain multiple skills sequentially)
    ├── CommandExecutor    (run local commands / scripts)
    └── OpenClawBridge     (forward to OpenClaw agent)
```

### 19-01: Skill Config Schema + Executor Interface

Agent owners define skills in `~/.agentbnb/skills.yaml` (NOT code — config):

```yaml
# ~/.agentbnb/skills.yaml
skills:
  # Mode A: API Wrapper — config-driven API calls, no code needed
  - id: tts-elevenlabs
    type: api
    name: "ElevenLabs TTS"
    provider: elevenlabs
    endpoint: "https://api.elevenlabs.io/v1/text-to-speech/{voice_id}"
    method: POST
    auth:
      type: bearer
      token: ${ELEVENLABS_API_KEY}  # from environment variable
    input_mapping:
      text: body.text              # map AgentBnB input → API param
      voice_id: path.voice_id
    output_mapping:
      audio: response.audio        # map API response → AgentBnB output
    pricing:
      credits_per_call: 5

  # Mode B: Pipeline — chain multiple skills
  - id: video-production
    type: pipeline
    name: "AI Video Pipeline"
    steps:
      - skill_id: text-gen-gpt4o    # Step 1: generate script
        input_mapping:
          prompt: "Write a 30-second video script about: ${params.topic}"
      - skill_id: tts-elevenlabs    # Step 2: voiceover from script
        input_mapping:
          text: ${prev.result.text}  # pipe previous step's output
      - skill_id: video-gen-kling   # Step 3: generate video
        input_mapping:
          prompt: ${steps[0].result.text}
      - command: "ffmpeg -i ${steps[2].result.video} -i ${steps[1].result.audio} -c:v copy -c:a aac output.mp4"
    pricing:
      credits_per_call: 40

  # Mode C: OpenClaw Bridge — forward to OpenClaw agent
  - id: creative-director
    type: openclaw
    name: "Creative Director"
    agent_name: chengwen-openclaw
    channel: telegram  # or webhook, or message_bus
    pricing:
      credits_per_call: 20

  # Mode D: Command — run a local script/binary
  - id: image-comfyui
    type: command
    name: "ComfyUI Image Gen"
    command: "python3 /path/to/comfyui_api.py --prompt '${params.prompt}'"
    output_type: file  # result is a file path
    pricing:
      credits_per_call: 15
```

### 19-02: API Executor (Mode A)

Config-driven HTTP API calls. No custom handler code needed.

Features:
- Support for REST APIs (GET/POST/PUT/DELETE)
- Auth types: bearer token, API key header, basic auth
- Input mapping: map AgentBnB params → API request (body, query, path, header)
- Output mapping: extract result from API response
- Retry logic: configurable retries on 429/500/503
- Timeout: configurable per skill
- Environment variable expansion: `${ENV_VAR}` syntax

Test: Create a mock HTTP server, configure a skill.yaml entry pointing to it, verify execute() calls the right endpoint with right params and returns mapped output.

### 19-03: Pipeline Executor (Mode B)

Chain multiple skills with dependency piping.

Features:
- Sequential step execution
- `${prev.result}` to reference previous step's output
- `${steps[N].result}` to reference any step's output
- `command` steps for local shell execution (ffmpeg, imagemagick, etc.)
- Partial failure handling: if step N fails, roll back credits for steps N+1..end
- Variable interpolation utility at `src/utils/interpolation.ts` (shared with Phase 22-01)

Test: Define a 3-step pipeline in skills.yaml, verify steps execute in order with correct piped inputs.

### 19-04: OpenClaw Bridge (Mode C)

Forward requests to an existing OpenClaw agent.

Features:
- Translate AgentBnB `{ skill_id, params }` → OpenClaw task format
- Support Telegram, webhook, and direct process communication
- Wait for OpenClaw agent response (with timeout)
- Map OpenClaw result back to AgentBnB format

Test: Mock an OpenClaw agent endpoint, verify bridge sends correct format and receives result.

### 19-05: Command Executor (Mode D)

Run local commands/scripts.

Features:
- Shell command execution with parameter substitution
- Support for `output_type: json | text | file`
- Sandboxing: configurable allowed commands (security)
- Working directory: configurable per skill
- Timeout: kill process after N seconds

Test: Configure a command skill that runs `echo ${params.text}`, verify output.

### 19-06: Gateway Integration

Wire SkillExecutor into Gateway so it replaces `--handler-url`:

```typescript

```

- `src/gateway/server.ts` L192-198: `fetch(handlerUrl, ...)` → `runtime.skillExecutor.execute(skillId, params)`
- `src/runtime/agent-runtime.ts`: add `skillExecutor` property, initialize from skills.yaml on `start()`

After this:

- Incoming requests → auth → escrow → SkillExecutor.execute() → settle

- No more "Handler error"

## Phase 21: Cross-Machine Credit Resolution (Track A)

### Why This Is Priority #1

Each agent's credit balance lives in its own local SQLite. Provider can't verify requester's balance. Integration tests mask this with shared in-memory DB.

### Solution: Signed Escrow Receipt (ADR-001 Option D)

```
Requester flow:
1. Requester holds escrow in OWN local DB (debit balance)
2. Requester generates signed receipt: { requester, amount, timestamp, signature }
3. Requester sends receipt + request to Provider's Gateway

Provider flow:
4. Provider verifies receipt signature (using requester's public key)
5. Provider trusts the receipt (requester has committed credits locally)
6. Provider executes the capability
7. Provider records earning in OWN local DB (credit balance)
8. On failure: Provider sends "release" message → Requester releases escrow locally
```

### 21-01: Agent Keypair Generation

On `agentbnb init`, generate an Ed25519 keypair:
- Private key: `~/.agentbnb/private.key` (never transmitted)
- Public key: published as part of Capability Card metadata

```typescript
// src/credit/signing.ts
function generateKeyPair(): { publicKey: Buffer; privateKey: Buffer };
function signEscrowReceipt(receipt: EscrowReceipt, privateKey: Buffer): string;
function verifyEscrowReceipt(receipt: EscrowReceipt, signature: string, publicKey: Buffer): boolean;
```

### 21-02: Signed Escrow Receipt Protocol

```typescript
interface EscrowReceipt {
  requester_owner: string;
  requester_public_key: string;
  amount: number;
  card_id: string;
  skill_id?: string;
  timestamp: string;  // ISO 8601
  nonce: string;      // prevent replay
  signature: string;  // Ed25519 signature of all above fields
}
```

### 21-03: Gateway Credit Verification Update

Change Gateway credit check from "query local DB" to "verify signed receipt":

```typescript
// Before (broken for P2P):
const balance = ledger.getBalance(requester);  // queries own DB — requester not there
if (balance < cost) throw new Error('Insufficient credits');

// After (works for P2P):
const receipt = request.escrow_receipt;
const valid = verifyEscrowReceipt(receipt, receipt.signature, requester_public_key);
if (!valid) throw new Error('Invalid escrow receipt');
if (receipt.amount < cost) throw new Error('Insufficient escrow');
// Trust the receipt — requester has committed credits locally
```

**Backward compatible**: If receipt present → verify signature. If absent → fallback to local DB check (same-machine scenario).

Key modification: `src/gateway/server.ts` L166-185

### 21-04: Settlement Protocol

After successful execution:
- Provider records `+N credits` in own ledger (earned)
- Requester settles escrow in own ledger (confirm debit)

After failure:
- Provider sends release signal
- Requester releases escrow in own ledger (refund)

Key modification: `src/credit/escrow.ts`

## GSD Discussion Decisions (pre-answered)

**SkillExecutor design**: Config-driven via skills.yaml. Four execution modes: API Wrapper, Pipeline, OpenClaw Bridge, Command. No custom handler code needed for API skills.

**Credit P2P solution**: Signed Escrow Receipts (ADR-001 Option D). Ed25519 keypair per agent. Requester holds escrow locally, sends signed receipt to provider. Provider verifies signature and trusts the receipt.

**Conductor scope**: v3.0 = Dumb Orchestrator (hardcoded templates). v4.0 = Smart Orchestrator (LLM decomposition) + Learning Orchestrator.

**Conductor identity**: Runs as a regular AgentBnB agent on the network. Not a platform feature. Charges 5 cr orchestration fee per task. Can be competed with (ADR-013).

**Parallel strategy**: Git worktree. Track A (main) and Track B (conductor-core) develop simultaneously with zero file conflicts. Merge after both complete, then Phase 22 integrates.

**Deployment**: Fly.io + DNS + Cloudflare Tunnel + GitHub Actions. Folded from v2.3's unstarted Phase 19 into v3.0 Phase 23.

**Agent philosophy reminder**: Every feature must pass "Does this require human intervention? If yes, redesign." The SkillExecutor specifically addresses this — agent owners define skills via YAML config, not by writing HTTP servers.
