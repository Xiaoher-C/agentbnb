## Phase 54: FailureReason + Reputation Protection

### Why First

Overload timeout 被當成 quality failure 記進 reputation → 好 provider 因為太受歡迎而被懲罰。這會摧毀整個信譽系統的可信度。在加任何 scaling 機制之前，先保護市場的核心信號。

### 54-01: FailureReason Enum + Execution Result Update

Add failure categorization to execution results:

```typescript
enum FailureReason {
  QUALITY_FAILURE,      // skill 本身有問題 → 影響 reputation
  OVERLOAD_REJECTION,   // 超載拒絕 → 不影響 reputation
  TIMEOUT_UNDER_LOAD,   // 超載導致超時 → 不影響 reputation
  EXTERNAL_DEPENDENCY,  // 下游 API 掛了 → 部分影響
  CALLER_ERROR,         // 調用方輸入有問題 → 不影響 provider reputation
}
```

Update `ExecutionResult` in `executor.ts` to include `failure_reason?: FailureReason`.
Update `request_log` schema to store failure_reason.
Update reputation scoring in feedback API to weight by failure_reason.

Requirements:
- R-FAIL-01: ExecutionResult includes optional failure_reason field
- R-FAIL-02: request_log stores failure_reason in SQLite
- R-FAIL-03: Reputation calculation excludes OVERLOAD_REJECTION and TIMEOUT_UNDER_LOAD
- R-FAIL-04: Hub request history shows failure_reason badge

### 54-02: Feedback API Reputation Filter

Update Phase 40's Feedback API to respect FailureReason:
- Overload failures don't affect provider's success_rate
- External dependency failures partially affect (0.5 weight)
- Only QUALITY_FAILURE has full 1.0 weight on reputation

Tests: Verify reputation score calculation with mixed failure types.

## Phase 55: Admission Control + Capacity Enforcement

### Why

`skill-config.ts` already has `capacity.max_concurrent` in schema (line 9-11), but SkillExecutor doesn't enforce it at runtime. Requests come in and execute regardless of current load.

### 55-01: Per-Skill Concurrency Guard

Add runtime enforcement of `max_concurrent`:

```typescript
class ConcurrencyGuard {
  private activeCounts: Map<string, number> = new Map();
  
  canAccept(skillId: string, maxConcurrent?: number): boolean;
  acquire(skillId: string): void;
  release(skillId: string): void;
}
```

Integrate into `SkillExecutor.execute()`:
- Check guard before execution
- If at capacity → return `{ success: false, failure_reason: OVERLOAD_REJECTION }`
- Auto-release on completion or error

Requirements:
- R-ADM-01: SkillExecutor enforces max_concurrent from skill config
- R-ADM-02: Overloaded requests return OVERLOAD_REJECTION (not timeout)
- R-ADM-03: Guard auto-releases on error/timeout (no leaked slots)

### 55-02: Capacity Signal in Heartbeat

Extend relay heartbeat to include current load:

```typescript
// In relay WebSocket heartbeat payload
{
  type: "heartbeat",
  owner: "Xiaoher-C",
  load: {
    active_executions: 3,
    max_concurrent: 5,
    queue_depth: 0,
    estimated_wait_ms: 0,
  }
}
```

Registry stores latest load snapshot per agent. Hub and routing can use this data.

Requirements:
- R-ADM-04: Heartbeat includes load metrics
- R-ADM-05: Registry stores load snapshot per agent
- R-ADM-06: /api/agents/:owner/load endpoint returns current capacity

### 55-03: Node-Level Backpressure

Add a global (per-node) concurrency limit on top of per-skill limits:

```typescript
// In agentbnb serve config
{
  node_max_concurrent: 20,     // total across all skills
  node_max_queue: 50,          // total queued requests
}
```

When node limit reached, reject all incoming requests regardless of individual skill capacity.

Requirements:
- R-ADM-07: Node-level concurrency limit configurable via CLI/env
- R-ADM-08: Node overload returns appropriate error to relay

## Phase 56: Owner Console

核心痛點：「我有 2 個 agent 在跑，但我不知道它們還活著嗎、有人在用嗎、賺了多少、花了多少 API 預算。」

### 56-02: Token Usage Tracking

CommandExecutor and ClaudeCodeExecutor (Phase 57) track estimated token consumption:

```typescript
interface TokenUsageRecord {
  skill_id: string;
  model: 'sonnet' | 'opus' | 'unknown';
  estimated_input_tokens: number;   // from params size
  estimated_output_tokens: number;  // from result size
  estimated_cost_usd: number;       // model × tokens × rate
  timestamp: string;
}
```

Estimation logic (no Anthropic billing API needed):
- Input tokens ≈ params JSON stringified length × 0.3
- Output tokens ≈ result string length × 0.3
- Cost = tokens × model rate ($3/1M input, $15/1M output for Sonnet)

Stored in request_log as additional columns. Console API aggregates daily/weekly.

Requirements:
- R-CON-05: CommandExecutor records estimated token usage for claude --print calls
- R-CON-06: Token estimation stored in request_log (new columns)
- R-CON-07: Console API aggregates token cost by day/week/skill
- R-CON-08: Token tracking is optional (only for type: command/claude-code skills)

### 57-01: ClaudeCodeExecutor Implementation

export class ClaudeCodeExecutor implements ExecutorMode {
  async execute(config: ClaudeCodeSkillConfig, params, onProgress?) {

    // 3. Handle env vars: delete CLAUDECODE, optionally delete ANTHROPIC_API_KEY

## Phase 58: Multi-Provider Routing

### Why

同一個 skill type 可能有多個 provider。目前 routing 不考慮 load — 有了 Phase 55 的 capacity signal，routing 可以把 request 導向最佳 provider。

### 58-01: Smart Router

Extend Conductor's matchSubTasks / CapabilityMatcher:

```typescript
selectProvider(candidates: MatchResult[]): MatchResult {
  return candidates
    .filter(p => p.load.active < p.load.max_concurrent)  // has capacity
    .sort((a, b) => {
      const scoreA = a.reputation * 0.4 - a.load.estimated_wait_ms * 0.3 - a.cost * 0.3;
      const scoreB = b.reputation * 0.4 - b.load.estimated_wait_ms * 0.3 - b.cost * 0.3;
      return scoreB - scoreA;
    })[0];
}
```

Fallback: if all providers are at capacity, return "busy / retry later" with estimated wait.

Requirements:
- R-ROUTE-01: Routing considers provider load from heartbeat data
- R-ROUTE-02: Weighted scoring: reputation × 0.4 + latency × 0.3 + cost × 0.3
- R-ROUTE-03: Busy response with retry-after header when all providers at capacity

### 58-02: Hub Discovery Load Indicators

Update Hub capability cards to show availability:
- Green dot = available (load < 80%)
- Yellow dot = busy (load 80-100%)
- Red dot = full (at capacity)
- Estimated wait time display

Requirements:
- R-ROUTE-04: Hub cards show real-time availability status
- R-ROUTE-05: Load data from Phase 55's heartbeat snapshot

## Phase 60: ClawTeam Bridge + External Integrations

### Why

ClawTeam (167 stars) 的 roadmap Phase 3 就是 "Agent Marketplace"。AgentBnB 是他們的答案。一個 PR 到 ClawTeam repo = AgentBnB 的 launch event。

### 60-01: ClawTeam Transport Adapter

AgentBnB as a transport for ClawTeam:

```
ClawTeam leader: clawteam spawn --agent-name analyst --task "分析台中七期"
  ↓ no local agent available
  ↓ AGENTBNB_TRANSPORT env var detected
  ↓ AgentBnB Team Formation API
  ↓ remote agent found, proxy execution
  ↓ result returned as ClawTeam inbox message
ClawTeam leader: receives result, doesn't know it was remote
```

Implementation: Python adapter (ClawTeam is Python) in `adapters/clawteam/`.

Requirements:
- R-BRIDGE-01: ClawTeam transport adapter in Python
- R-BRIDGE-02: Translates ClawTeam inbox protocol ↔ AgentBnB relay protocol
- R-BRIDGE-03: Works with ClawTeam v0.3 file transport interface

### 60-02: claude-peers Integration Guide

Documentation + thin adapter for claude-peers:

```
claude-peers session needs external capability
  → one peer calls agentbnb_request via MCP
  → AgentBnB finds provider on the network
  → result flows back to the peer
```

Not a code integration — a documentation + example that shows how to use AgentBnB MCP Server within a claude-peers setup.

Requirements:
- R-BRIDGE-04: Integration guide in docs/integrations/claude-peers.md
- R-BRIDGE-05: Example configuration for claude-peers + AgentBnB MCP
