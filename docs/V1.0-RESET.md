# AgentBnB V1.0 Reset

A short note on why the public version is V1.0 even though the codebase has shipped through v3, v4, ... v9.

## What V1.0 means

V1.0 is the first coherent public version of AgentBnB. The product is one thing: a P2P protocol where agents publish capabilities, request capabilities from other agents, and settle in credits. The identity stack, escrow, conductor, hub, CLI, MCP server, and relay are all unchanged. Only the version surface is simpler.

## Why a reset

The internal version had crept to 9.2.2 across many milestones (SkillExecutor, Conductor, Signed Escrow, Relay, Agent Economy, Genesis Flywheel, OpenClaw Hardening, Team Formation, Identity Convergence, Identity Protocol). Each was a real shipping milestone. None of them were the product.

A new reader landing on a "v9.2.2" project reasonably assumes either eight prior major versions of breakage, or a long history of public iteration they need to catch up on. Neither is the actual story. The story is: one product, built up over many internal milestones, now stable enough to call V1.0.

A coherent first impression matters more than version-number continuity.

## What did NOT change

- Code — no module renamed, removed, or restructured for the reset
- Tests — 1,800+ existing tests still run
- The live relay
- The credit ledger and escrow system
- The v9 identity stack (DID + UCAN + Verifiable Credentials)
- The OpenClaw plugin
- The MCP server (6 tools)
- The framework adapters (LangChain / CrewAI / AutoGen)

The reset is a version-string and framing change. Nothing under `src/` was touched for it.

## Where the prior history lives

Nothing was deleted. The pre-reset history is preserved in:

- [`CHANGELOG.md`](../CHANGELOG.md) — full release history from v1.1 through v9.x
- [`docs/AGENT-IDENTITY-PROTOCOL.md`](AGENT-IDENTITY-PROTOCOL.md) — the v9 identity protocol design doc
- [`docs/cutovers/`](cutovers/) and [`docs/maintenance/`](maintenance/) — operational history
- Phase plans and verification artifacts from earlier milestones remain in the repo for anyone tracing how a given subsystem was built

If you want to know what shipped before V1.0, start at `CHANGELOG.md`.

## Going forward

V1.0 is the public version. Future releases follow semver from this baseline: V1.1 for additive changes, V2.0 for breaking ones.

Internal milestone naming will continue, because it's useful for planning, but it won't be surfaced in `package.json` or the README. If you read about "milestone v10" or a "v11 line" in planning docs, that's an internal track. The public version is whatever `package.json` says.
